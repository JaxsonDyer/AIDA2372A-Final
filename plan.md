End-to-End MLOps Pipeline for Employee Attrition Prediction
Jaxson Dyer
1. Introduction
Project Goal: Build and operationalize a classification model that predicts whether an employee is likely to leave the company, enabling proactive HR intervention.
Context & Rationale: Employee turnover is one of the most costly and disruptive challenges a growing company faces. Replacing a single employee can cost upwards of 50-200% of their annual salary when accounting for recruiting, onboarding, and lost productivity. The end-users of this system are HR managers and business leaders who need timely, data-driven signals about flight risk so they can act before it is too late.
A one-off notebook model is insufficient here for several reasons. Workforce demographics shift over time, company culture evolves, and external economic conditions change, all of which affect attrition patterns. A model trained once will degrade in accuracy over time. A robust MLOps pipeline ensures the model is continuously retrained on fresh data, tracked for performance, and redeployed automatically without requiring manual effort from a data scientist each time.
Scope & Success: The pipeline will cover data versioning with DVC, experiment tracking with MLflow, model serving with Flask and Docker, automation with GitHub Actions, and orchestration with Apache Airflow. The project is considered successful when a change to the training script automatically triggers a CI/CD pipeline that lints, tests, builds, and pushes a new versioned Docker image with no manual steps required.
2. Data Versioning & Baseline Model
Dataset: The IBM HR Analytics Employee Attrition dataset, publicly available on Kaggle. It contains 1,470 employee records with 35 features including age, job role, salary, overtime status, and a binary attrition label.
DVC Strategy: The raw CSV will be added to DVC tracking rather than committed directly to Git. DVC will be configured with a remote storage location (Google Drive for simplicity) so that any collaborator can reproduce the exact dataset by running a DVC pull command after cloning the repository. The .dvc file committed to Git stores a hash of the data file, meaning the dataset is fully versioned and reproducible without bloating the repository. This is superior to committing raw data to Git because large files slow down the repository, cannot be efficiently diffed, and make it impossible to track data changes independently of code changes.
Baseline Model: Logistic Regression will serve as the baseline. It is interpretable, computationally cheap, and provides a meaningful performance floor against which more complex models can be compared. Given the class imbalance in this dataset (roughly 84% of employees did not leave), F1 score will be the primary evaluation metric rather than raw accuracy.
3. Experiment Tracking with MLflow
Experiment Design: Four experiments will be run in sequence. First, the Logistic Regression baseline. Second, a Random Forest with default hyperparameters. Third, a tuned Random Forest where the number of estimators and maximum tree depth are varied across multiple runs. Fourth, an XGBoost classifier, which tends to perform well on tabular classification problems with class imbalance.
MLflow Integration: Every training run will log the model algorithm name, all hyperparameters, training and validation F1 score, accuracy, and the serialized model artifact. Logging artifacts alongside metrics is important because it links the exact model file to the exact conditions that produced it, making any run fully reproducible.
Model Selection: The MLflow UI will be used to sort all runs by validation F1 score. The best performing run will be registered in the MLflow Model Registry and promoted to the Production stage. A screenshot of the MLflow comparison table will be included as evidence.
4. Model Packaging & Containerization
Model Serialization: The winning model artifact will be retrieved from the MLflow run and saved locally for use in the API.
API Design: A Flask application will expose a prediction endpoint that accepts a JSON payload of employee features, loads the serialized model on startup, runs inference, and returns a prediction of 0 (stays) or 1 (leaves) along with a probability score.
Containerization Strategy: A Dockerfile will package the Flask app and its dependencies into a self-contained image based on a slim Python base image to keep the image size minimal. The image will be built and pushed to Docker Hub as part of the CI/CD pipeline, ensuring that any environment can run the exact same prediction service by pulling the image.
5. CI/CD with GitHub Actions
Automation Strategy: Every push to the main branch will trigger an automated workflow. The goal is that no broken code can reach the Docker image that serves predictions.
Workflow Design: The pipeline will consist of three sequential jobs. The first job installs dependencies, runs a linter to enforce code style, and runs unit tests to verify the API logic. The second job, which only runs if the first succeeds, builds the Docker image. The third job pushes the image to Docker Hub using credentials stored as GitHub Secrets.
Proof of Automation: A GitHub Actions status badge will be embedded at the top of the README, and a screenshot of a successful workflow run will be included in the notebook.
6. Workflow Orchestration with Apache Airflow
Why Airflow over a Cron Job: A cron job can schedule a script to run, but it offers no visibility into failures, no dependency management between tasks, no retry logic, and no historical run logs. Airflow provides all of these through its UI and DAG framework, making it far more suitable for a production retraining pipeline.
DAG Design: The retraining DAG will run on a weekly schedule and contain four tasks in sequence. The first task pulls the latest version of the dataset using DVC. The second task runs the training script to produce a new model. The third task logs the new run to MLflow. The fourth task compares the new model's F1 score against the currently registered production model and only promotes it if performance has improved, preventing a bad retrain from replacing a good model.
Evidence: A screenshot of the Airflow Graph View showing all four tasks completing successfully on a manual trigger run will be included in the notebook.
7. Discussion
Challenges & Learnings: The most anticipated challenges are configuring DVC remote authentication, managing Python dependency conflicts between MLflow, scikit-learn, and XGBoost inside Docker, and getting Apache Airflow running locally given its significant resource overhead. The main takeaway from this project is that the tooling around a model often requires more engineering effort than the model itself, which is precisely the point of MLOps as a discipline.
Limitations: The current pipeline lacks automated data validation, meaning a corrupted or schema-changed dataset could silently cause a bad model to be trained. There is also no monitoring of the live prediction API, so model drift in production would go undetected. There is no A/B testing framework either to safely roll out new model versions before full deployment.
Future Work: Two high-impact improvements would be integrating Great Expectations to perform automated data quality checks as the first step of the DAG, rejecting any dataset that fails schema or distribution checks before training begins. The second would be deploying Prometheus and Grafana to monitor API response times and track prediction distribution over time, enabling early detection of data drift and model degradation.
